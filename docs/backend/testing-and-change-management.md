# Testing And Change Management

## 목적

- 이 문서는 가드레일이 문서로만 남지 않도록 테스트와 변경 관리 기준을 정의합니다.
- 규칙 위반을 구현 점검과 테스트에서 조기에 잡는 것이 목적입니다.

## 책임과 경계

- 테스트는 레이어별 책임을 증명해야 합니다.
- 문서 변경은 코드 변경과 분리되지 않습니다.
- 예외 허용은 일회성 구두 합의가 아니라 기록 가능한 절차를 따라야 합니다.

## Must

- MUST DTO 테스트는 검증 규칙과 허용/거부 입력을 증명합니다.
- MUST Controller 테스트는 Guard, 상태코드, 파라미터 위임, 문서 계약에 맞는 transport 진입점 동작을 증명합니다.
- MUST Service 테스트는 정합성 검증, 예외 변환, dependency orchestration을 증명합니다.
- MUST integration 테스트는 전역 `ValidationPipe`, 응답 인터셉터, 예외 필터가 실제로 함께 동작하는지 확인합니다.
- MUST 새 모듈이나 새 레이어 패턴을 도입하면 관련 문서와 체크리스트를 같이 업데이트합니다.
- MUST 가드레일 변경은 “왜 기존 규칙이 부족했는가”를 PR 설명에 적습니다.
- SHOULD 자주 어기는 규칙은 테스트, 스캐폴딩, 구현 점검 체크리스트로 자동화합니다.

## Must Not

- MUST NOT 테스트가 레이어 경계를 무시한 채 구현 세부사항에만 결합됩니다.
- MUST NOT 문서 변경 없이 코드에서만 새로운 예외 패턴을 퍼뜨립니다.
- MUST NOT 임시 규칙을 문서에 본 규칙처럼 남기고 종료 조건을 생략합니다.

## Review Questions

- 변경된 코드가 어느 레이어 책임을 추가하거나 이동시켰는가?
- 해당 책임 이동을 증명하는 테스트가 있는가?
- 문서와 체크리스트가 새 패턴을 반영하고 있는가?
- 예외 허용이 반복될 가능성이 있다면 문서에 승격되었는가?

## Exception Process

- 긴급 패치로 문서 동기화가 늦어질 수는 있지만, 같은 PR 또는 즉시 후속 PR에서 보완합니다.
- 새 예외가 한 번 이상 반복되면 문서와 템플릿에 반영합니다.
- 테스트가 당장 어렵다면 왜 어려운지, 어느 레이어에서 보완할지 PR에 남깁니다.
